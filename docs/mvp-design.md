# Codex Workbench MVP Design

## 1. 产品定位

这不是一个“在 Obsidian 里再放一个聊天网站”的插件。

它更像是一个编辑工作台，目标是让用户在写作、整理、批注时，直接围绕当前笔记和当前选区发问，并且把回答快速回写到文档里。

## 2. 设计原则

### 编辑优先

用户最重要的动作不是“聊天”，而是：

- 看一段话
- 选中它
- 发问
- 把结果改回笔记

### 上下文要明确

AI 面板里必须能清楚看到当前上下文来自哪里：

- 哪个笔记
- 哪段选区
- 附近有什么文本

### 回写比回答更重要

除了生成文字，插件必须提供顺手的落笔动作：

- 插入到当前光标
- 替换当前选区
- 复制回答

### 原生而不平庸

视觉上跟随 Obsidian 主题，但不做空白、廉价、千篇一律的聊天框。

## 3. MVP 功能范围

### 包含

- 右侧 `Codex Workbench` 面板
- 聊天输入与会话流
- 读取当前编辑器选区
- 选区附近浮动 `Ask Codex` 按钮
- 编辑器右键菜单里的 `Ask Codex about selection`
- 基础设置页
- 本地 `codex app-server` 会话接入
- 持久化恢复同一个 Codex thread
- `workspace-write` 和 approvals 暴露
- Mock 模式和可配置 HTTP 接口
- 将最近回答插入 / 替换 / 复制

### 不包含

- 全库检索和向量数据库
- 多轮会话的跨笔记长期记忆
- 引文定位与文献管理
- 流式输出
- 多模型切换面板

## 4. UI 架构

### 右侧面板

从上到下分为四层：

1. 顶部状态栏
   - 插件标题
   - 当前 provider 状态
   - 快捷动作按钮
2. 当前上下文卡片
   - 当前笔记名
   - 当前选区摘要
   - 清空上下文
3. 会话流
   - 用户气泡
   - 助手气泡
   - 时间和角色元信息
4. 底部输入区
   - 多行输入框
   - 发送按钮
   - 小提示文案

### 编辑区浮动入口

当用户在编辑器里选中文本时，在选区右上侧出现一个小型悬浮按钮：

- 文案：`Ask Codex`
- 作用：直接把当前选区送入右侧工作台

这个入口是加速器，不是唯一入口，所以同时保留右键菜单和命令面板。

## 5. 视觉风格

### 关键词

- editorial
- annotation
- workbench
- tactile

### 颜色

基于 Obsidian 主题变量，只加一层温暖的品牌感：

- 主强调色：琥珀橙
- 辅助色：深蓝灰
- 成功状态：沿用系统绿

### 形态

- 面板头部采用窄条渐变，强化辨识度
- 上下文卡片像批注夹层，不做大面积纯色块
- 气泡边角不完全圆润，保留一些“工具感”

## 6. 核心交互流

### 场景一：划线提问

1. 用户在笔记中选中一段文本
2. 浮动按钮出现
3. 点击后自动打开右侧面板
4. 面板里生成一条用户问题，并附带选区上下文
5. 助手返回回答
6. 用户决定插入、替换或复制

### 场景二：从面板主动提问

1. 用户直接在右侧输入问题
2. 如果当前有选区，自动附带选区
3. 如果没有选区，则附带当前笔记元信息
4. 返回结果后可回写

## 7. 技术实现策略

### UI

- 不引 React
- 使用 Obsidian 插件原生 DOM API
- 用少量 CSS 变量做风格层

### 数据流

- 会话先保存在内存
- 最近回答缓存为单独状态，便于回写
- 上下文按“当前活动编辑器”实时采集

### 网络

- 默认 `Local Codex app-server`
- 保持一个本地 Codex thread/session
- 使用 `turn/start` 获取流式消息增量
- 保存 `threadId` 并在下次打开插件时 `thread/resume`
- 用本地审批弹窗承接 command / file-change / permissions 请求
- `Mock` 和 HTTP provider 作为兼容后备
- 支持 `OpenAI-compatible`
- 支持简单 `Generic JSON`

## 8. 后续演进

- 标题级上下文与整篇上下文切换
- Prompt Presets
- 流式输出
- 针对选区的批注锚点
- 历史会话按笔记分组保存
